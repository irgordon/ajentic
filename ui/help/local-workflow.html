Local workflow

The local workflow moves in one visible path: provider setup → constrained invocation → provider output pipeline → validation/review → staged proposal → staged validation → candidate review → operator decision → local candidate materialization → replay/status → evidence export → session package → restore/history.

What each stage does

  • Provider setup records a local adapter shape and provider configuration. It protects the shell from unknown provider paths.
  • Constrained invocation runs only the allowed local path. It prevents arbitrary commands, cloud paths, and secret use.
  • The provider output pipeline displays output as untrusted material. It prevents provider output from becoming authority by itself.
  • Validation and review check shape, linkage, policy status, and review notes. They protect the operator from treating raw output as candidate output.
  • A staged proposal prepares possible candidate material. It prevents direct promotion from provider output.
  • Staged validation checks the staged proposal before any local decision is recorded.
  • Candidate review shows what the operator is deciding on.
  • Operator decision records local intent only. It does not authorize actions.
  • Local candidate materialization creates local candidate output after the required local steps. It is not production approval.
  • Replay/status summarizes recorded local decisions. It does not repair replay.
  • Evidence export and session package collect local evidence for review. Evidence is not authority.
  • Restore/history shows read-back and restore previews. Verification does not repair replay or promote recovery.

This local workflow is local-only and non-production.
