## 3. Boundary audit

### Rust authority

- Rust remains authority for lifecycle, governance, promotion, ledger, audit, replay, and validation surfaces (`core/src/candidate`, `core/src/governance`, `core/src/ledger`, `core/src/audit`, `core/src/replay`, `core/src/domain`).
- Promotion to `PromotedTier1` is enforced through governance promotion logic (`core/src/governance/promotion.rs`) and lifecycle transition checks reject invalid direct promotion from `Created`.
- Replay and audit derive from ledger facts only; no provider/UI/script authority leaks detected.

### Python adaptation

- Python adapters remain non-authoritative and emit line protocol only.
- OpenAI adapter requires `AJENTIC_OPENAI_API_KEY`, applies timeout and output byte checks, and line-escapes output via shared helper.
- Ollama adapter enforces timeout/output checks and escapes line output, but uses a **local** `_line_value` and local `REQUIRED_KEYS` instead of shared protocol helpers; this is consistency drift, not authority drift.
- No promotion/lifecycle/ledger/audit/replay authority in Python.

### UI visibility

- UI is static visibility-only; no network/subprocess/provider/backend calls found.
- Renderer escapes interpolated values before `innerHTML` insertion.
- Sample state keeps denied promotion with lifecycle `Rejected` (no contradictory `PromotedTier1` state).
- UI docs maintain non-authoritative stance and clarify Tier-1 review semantics.

### Bash glue

- Shell scripts use `#!/bin/sh` and `set -eu`.
- Scripts are thin wrappers/checks and do not encode governance/promotion/ledger/replay policy logic.
